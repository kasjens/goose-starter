# Using Goose with a GitHub Enterprise Copilot Seat

If your organization uses **GitHub Enterprise (GHE.com)** and gives you a Copilot
seat there, you can point Goose at your enterprise tenant instead of public
`github.com`. This keeps authentication and model traffic inside your tenant.

Replace `your-company.ghe.com` below with your own enterprise host.

> **Note:** This uses your **company-backed GitHub Enterprise Copilot** â€” an
> officially provisioned enterprise seat, not a personal or unofficial one.
> Authentication and model traffic stay inside your enterprise tenant (and its
> region), which is exactly why the enterprise host is used here rather than
> public `github.com`. As with any tool at work, follow your organization's usage
> policies; in regulated environments (e.g. GxP) confirm your team's guidance
> before using it for work code.

## What needs to be configured

There are two pieces of Goose configuration plus a manual sign-in:

1. An environment variable that tells Goose which host to authenticate against.
2. (Optional) Pinning a default Copilot model in Goose's config file.
3. A one-time device-flow sign-in (interactive â€” cannot be scripted).

## 1. Point Goose at your enterprise host

Set the `GITHUB_COPILOT_HOST` environment variable **before** your first Copilot
sign-in, so the device flow targets your tenant and not public `github.com`.

**Windows (PowerShell):**

```powershell
[Environment]::SetEnvironmentVariable('GITHUB_COPILOT_HOST', 'your-company.ghe.com', 'User')
```

**macOS / Linux (bash/zsh):** add to your shell profile (`~/.zshrc`, `~/.bashrc`):

```bash
export GITHUB_COPILOT_HOST="your-company.ghe.com"
```

Restart Goose (fully quit and relaunch) so it picks up the variable.

> Use the bare host only â€” no `https://` scheme and no trailing slash.

## 2. Connect the Copilot provider (device-flow sign-in)

This step is interactive and must be done by hand:

1. Fully quit and relaunch Goose so it reads `GITHUB_COPILOT_HOST`.
2. Go to **Provider settings â†’ GitHub Copilot** and start the sign-in.
3. It should send you to `https://your-company.ghe.com/login/device`.
   Sign in with your **managed enterprise account**.
4. The verification code is generated by Goose (shown in-app / copied to
   clipboard).
5. If the Desktop UI won't surface the code, use the CLI instead â€” it honors the
   same environment variable:

   ```bash
   goose configure
   ```

## 3. Pin a default model (optional)

Goose stores its config in a `config.yaml` file:

- **Windows:** `%APPDATA%\Block\goose\config\config.yaml`
- **macOS:** `~/.config/goose/config.yaml`
- **Linux:** `~/.config/goose/config.yaml`

The relevant keys are:

```yaml
active_provider: github_copilot
providers:
  github_copilot:
    enabled: true
    model: claude-opus-4.8
    configured: true
```

Notes:

- GitHub Copilot uses **decimal** model ids, e.g. `claude-opus-4.8`,
  `claude-opus-4.7`, `gpt-5.5`.
- The `providers.github_copilot` block only exists **after** you've connected the
  Copilot provider at least once (step 2). Connect first, then edit.
- **Goose is closed while you edit.** Goose rewrites `config.yaml` on exit with
  the last-used model, so an edit made while it's running will be overwritten.

## 4. Keep the model from resetting

In the model picker, select your chosen model (e.g. **Claude Opus 4.8**) rather
than **Auto**. Goose saves the last-used model as the default on exit, so if you
leave it on **Auto** it will overwrite whatever default you pinned.

## Quick checklist

- [ ] `GITHUB_COPILOT_HOST` set to your enterprise host (no scheme/slash)
- [ ] Goose restarted after setting the variable
- [ ] Copilot provider connected via device flow against your tenant
- [ ] (Optional) default model pinned in `config.yaml` while Goose is closed
- [ ] Model picker set to a specific model, not **Auto**
- [ ] Usage follows your organization's policies (confirm guidance for regulated work)
